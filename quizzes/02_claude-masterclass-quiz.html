1MC
A content writer submits a prompt: "Write a blog post about AI trends — make it friendly but professional, conversational yet authoritative, casual but credible." Which prompt failure does this represent, and what is the correct fix?
AMistake 3 (no role). Fix: assign a specific expert role and target audience before stating style preferences.
BMistake 2 (wall of text). Fix: break the adjectives into separate labeled constraint sections using line breaks.
CTechnique 6 failure (no style example). Fix: replace the contradictory adjectives with one concrete paragraph of writing in the target voice.
DMistake 1 (searching). Fix: reframe as a brief with context about the intended platform and reader before listing tone preferences.
2MC
The four-block formula states the Task block should contain "one clear deliverable." A user writes: "Task: Write a welcome email AND suggest three subject line variations AND provide a short A/B test rationale." What specific failure does this create?
AClaude will refuse the request because the formula only allows single-sentence task specifications.
BMultiple asks in the Task block force Claude to prioritize between deliverables, often completing the first and giving diminishing quality to subsequent ones — reverting to averaging behavior.
CThe formula requires each deliverable to have its own XML tag; the failure is structural, not conceptual — wrapping each in a tag resolves it without separating the tasks.
DMultiple deliverables only cause problems in wall-of-text prompts; when the four-block formula is used, Claude handles multiple task items reliably.
3MC
A developer wants to generate 100 product descriptions in a consistent JSON format using Claude. They plan to use XML tags for structure. Which additional technique would most directly eliminate Claude's tendency to open with "I'll generate these product descriptions for you. Here's the format I'll follow:" before the actual output?
AConstraint stacking — add a rule like "no preamble or explanation before the output."
BPrefill — begin Claude's response with the opening character of the JSON array, so Claude continues generating JSON from word one without any preamble.
CMotivated instructions — explain why preamble is unwanted, since understanding the reason prevents Claude from defaulting to it.
DPrompt chaining — separate the format definition from the generation task across two turns so Claude commits to the format before generating content.
4MC
Prompt chaining's advantage over a single mega-prompt is strongest in Claude specifically because of a technical characteristic. Which characteristic, and why does it matter for chain coherence?
AClaude's response speed is faster per token than competitors, so chains complete before the model's attention degrades across long sessions.
BClaude's 200K context window keeps the entire chain in one conversation — step-one instructions carry full weight at step five, whereas shorter-context models lose chain coherence as earlier turns fade.
CClaude's XML training means chained outputs in XML format are more precisely parsed between steps, reducing semantic drift across multi-step chains.
DClaude's Extended Thinking mode automatically activates between chain steps, silently improving inter-step reasoning quality without user intervention.
5MC
An analyst writes: "Don't include speculative statements." Claude still includes some hedged claims. The analyst rewrites: "Don't include speculative statements because this report will be submitted to a regulatory body that requires all claims to be traceable to cited data." What technique is this, and why does the rewrite produce more compliant output?
AConstraint stacking — stacking a regulatory context alongside the rule increases the severity weighting Claude assigns to that constraint, producing stricter compliance.
BMotivated instructions — adding the rationale enables Claude to apply the spirit of the rule in edge cases, catching hedged claims the command-only version didn't flag.
CIterative refinement — the rewrite is effectively a second-pass revision prompt, triggering a self-critique loop that improves compliance.
DFour-block formula — the regulatory context belongs in the Context block, and moving it there prevents Claude from treating it as an instruction to respond to rather than background to apply.
6MC
In the iterative refinement technique, what specific instruction triggers Step 2, and what makes it more effective than simply asking Claude to "improve" the draft?
A"Improve this draft" — effective because it activates Extended Thinking mode automatically, producing a deeper revision than standard prompts.
B"Critically analyze your response. Find weak spots, redundancies, gaps, contradictions" — more effective because it assigns Claude the editor role explicitly, forcing systematic problem identification before rewriting rather than surface polishing.
C"Rewrite this with higher quality" — more effective because the open-ended instruction gives Claude latitude to apply its strongest revision strategies without constraint.
D"Apply the four-block formula to your own response" — restructures the draft's reasoning rather than polishing its surface, making it the most effective second-step trigger.
7MC
A routine developer receives no output after their remote routine fires. The session log shows: "Environment variable OPENAI_API_KEY not found. Attempting to load from .env... File not found. Exiting." What is the root cause and the correct fix?
AThe GitHub repo is missing a .env template. Fix: add a .env.example to the repo and reference it in the setup script field of the cloud environment.
BThe API key was stored in a .env file which is gitignored and never reaches the cloud environment. Fix: move the key to the Environment Variables panel in Cloud Environment, and add "do not look for a .env file" to the routine's instruction prompt.
CThe cloud environment's network trust mode is blocking the API endpoint. Fix: switch from "trusted" to "custom" and allowlist the external API domain.
DThe routine is set to "local" execution mode, which does not support environment variables. Fix: switch to remote mode so the cloud environment variables panel becomes available.
8MC
A team builds a routine that automatically replies to client emails every morning. Three weeks later, a team member goes on vacation. Which of the seven gotchas is now most relevant, and what should have been designed differently?
AGotcha 4 (statelessness) — the routine loses its reply history between runs. Fix: push sent-reply metadata to a GitHub file to persist state across sessions.
BGotcha 6 (identity scope) — all auto-replies send from the vacationing team member's account. The routine should have used a two-step architecture: Routine A drafts replies → human approves → Routine B sends.
CGotcha 7 (no mid-run approval) — the routine cannot pause to handle unusual emails. Fix: add a confidence-score condition in the prompt that routes ambiguous emails to a Slack flagging step instead of sending directly.
DGotcha 5 (claude/ branch guardrail) — the routine's email drafts are being pushed to the main branch, causing version conflicts with other team workflows.
9MC
A developer needs a routine to fire every 20 minutes when a specific Slack channel receives a message. Which trigger architecture should they use, and why?
ASchedule trigger set to "hourly" — the minimum supported interval is 20 minutes when using the Slack connector, making this the correct choice.
BGitHub Events trigger — configure it to fire on Slack channel messages, then filter by message content in the routine's instructions.
CAPI trigger — configure Slack to send a webhook to the routine's HTTPS endpoint on each message, and use an external tool like N8N or Zapier to handle the 20-minute debounce logic before posting to the endpoint.
DSchedule trigger with a custom cron expression for 20-minute intervals — cron expressions bypass the 1-hour minimum that applies only to the preset schedule options.
10MC
A solo founder launches Claude Design without filling in the design system first, generates five landing pages, and finds they all look identical and generic. What is the root cause, and which single action would most improve all future generations?
AThe high-fidelity mode was not selected — switching to high-fidelity produces brand-consistent outputs because it uses the full design token set rather than defaults.
BNo design system was defined — Claude defaulted to its generic aesthetic on every run. Defining a specific 200-word+ brand description, colors, fonts, and voice, then publishing it as default, ensures every future generation draws from the brand rather than Claude's defaults.
CThe prompts were too short — longer prompts with detailed styling instructions per element override Claude Design's default aesthetic even without a design system.
DThe Tweaks panel was not used after generation — running through all Tweaks options after each generation produces a distinct aesthetic even from a generic system.
11MC
After reviewing a generated landing page in Claude Design, a user wants to change just the hero section's headline font weight and the CTA button color. What is the most token-efficient approach?
ARegenerate the full page with an updated prompt that specifies the new font weight and button color — full-page regeneration ensures all elements update consistently.
BUse Markup to click the headline and give targeted font weight feedback, then use the Edit property panel to change the button color directly — zero-token Edit + minimal-cost Markup before any full regeneration.
CUse the Tweaks panel to adjust font weight and color — both properties are always available in Tweaks since it auto-generates controls for every design property.
DExport to Claude Code and make the changes there — code-level edits don't consume Claude Design tokens and allow precise property control.
12MC
What is the specific operational significance of the claude/ branch guardrail in remote routines, and what risk does disabling it introduce?
AThe guardrail enforces that all routine-generated code passes a syntax check before committing. Disabling it allows malformed code to enter the repository without review.
BBy default Claude can only push to branches prefixed with "claude/" and cannot touch main or release branches. Disabling it means a bad routine run could push directly to main — potentially shipping broken or unreviewed changes to production.
CThe guardrail limits how many commits Claude can make per run. Disabling it allows unlimited commits per run, which can bloat repository history with incremental routine outputs.
DWithout the guardrail, Claude merges its own branches automatically, removing the pull request step and bypassing required code reviews in team settings.

AExtended Thinking improves caching efficiency because the thinking block is deterministic — the same prompt always produces the same thinking trace, which the cache reuses on subsequent calls.
BExtended Thinking reduces caching efficiency because the thinking block changes on every run, even for identical prompts — fewer cache hits means higher token costs for repetitive identical workflows.
CExtended Thinking has no effect on caching because thinking blocks are processed server-side and are excluded from the cache key computation entirely.

14MC
A developer building a long-running code refactoring agent needs to choose between Sonnet 4.5 and Haiku 4.5 for the execution phase. The task involves 40+ tool calls and maintaining variable name consistency across a 200-file codebase. Which model and why?
AHaiku 4.5 — its 4–5× speed advantage compensates for any context tracking issues, and the cost savings across 40+ tool calls are significant enough to justify the occasional consistency error.
BSonnet 4.5 — Haiku loses track in long sessions and may forget variable names or change identifiers, which would be catastrophic in a cross-file refactoring task requiring consistency across 200 files.
CHaiku 4.5 — the sub-agent pattern (Sonnet orchestrates, Haiku executes) means context tracking is Sonnet's responsibility, not Haiku's, so Haiku's long-session weakness is not relevant.
DSonnet 4.5 or Haiku 4.5 equally — the claude.md memory file in Claude Code persists variable names across all sessions and tool calls, eliminating both models' long-session context limitations.
15MC
The effort parameter is exclusive to Opus 4.5. A financial analyst uses it set to "high effort" for quarterly reports and "low effort" for quick data lookups. What is the fundamental advantage of this approach over switching between Opus and Haiku for the same two task types?
AThe effort parameter is cheaper than Haiku at low effort settings, making Opus more cost-efficient than Haiku for quick lookups when effort is reduced.
BOne model with variable thoroughness eliminates model-switching overhead and maintains a consistent reasoning baseline — Opus at low effort still brings Opus-level reliability, avoiding Haiku's long-session context-tracking weakness.
CThe effort parameter enables Opus to access different pre-trained sub-models internally, each specialized for a thoroughness level — this specialization outperforms Haiku on speed at the low effort setting.
DSwitching between Opus and Haiku mid-session resets the context window; the effort parameter preserves context continuity across task types within the same session.
16MC
Step 3 of the prompt debug protocol states: "give the model permission for uncertainty." What specific failure mode does this instruction address, and what exact phrase is recommended?
AIt addresses over-refusal — Claude's tendency to decline tasks when it lacks full information. The phrase "you are permitted to use partial information to complete the task" unlocks responses Claude would otherwise refuse.
BIt addresses hallucination — Claude defaults to fabricating plausible-sounding information rather than admitting gaps. The phrase "if data is insufficient, say so, don't guess" dramatically reduces hallucinations.
CIt addresses verbosity — Claude over-explains when uncertain. The phrase "if you're not sure, keep it brief" reduces word count in uncertain sections without affecting confident sections.
DIt addresses scope creep — Claude expands beyond the task when it finds related material. The phrase "if the task exceeds the context, stop and ask for clarification" prevents out-of-scope additions.

BClaude 4.5 (Opus) — regulated industries, costly mistakes, and the need for auditable reasoning traces are the exact conditions where Claude's Extended Thinking transparency and dependability advantage over Gemini is most significant.
CGemini 3 Pro — its lower per-token cost ($2/$12 vs. $5/$25) makes it more viable for hundreds of daily analyses without the budget implications of Opus, while maintaining sufficient accuracy for compliance work.
DClaude 4.5 (Haiku) — at the volume required, cost-efficiency is paramount and Haiku's 73.3% SWE-Bench accuracy is sufficient for document flag detection without the overhead of Opus.
18MC
A user notices their Claude 4.5 session abandoned a complex data processing task mid-way with the message "I'm approaching my context limit and cannot continue." Which architecture change in Claude 4.5 was specifically designed to prevent this, and how does it work?
AThe memory tool — Claude externalizes processed data to a file mid-task, freeing context window space and allowing the task to continue without abandonment.
BContext awareness — Claude 4.5 models actively track remaining tokens after every tool call, preventing premature abandonment by knowing precisely how much space remains rather than conservatively estimating and giving up early.
CThe 200K context expansion — prior models abandoned tasks because their smaller context windows genuinely ran out; Claude 4.5's larger window eliminates abandonment by providing sufficient space for any realistic task.
DContext compaction — Claude 4.5 automatically summarizes completed task steps into a compressed representation, freeing context window space to continue the remaining steps without abandonment.

What does "interleaved thinking" mean in the context of Claude 4.5's Extended Thinking, and which Claude capability does it most directly power?
AClaude alternates between thinking and generating output within a single response — producing a more conversational, less structured answer than standard extended thinking mode.
BClaude reasons between tool calls — run a command, see the result, think about what to do next, run another command — creating coherent multi-step agentic behavior powered by reasoning loops.
CClaude alternates between Extended Thinking mode and standard response mode across a conversation session, preserving caching efficiency while still applying deep reasoning to selected turns.
DMultiple Claude models think simultaneously on the same problem and their reasoning traces are merged — a multi-agent thinking architecture exclusive to the Claude 4.5 family.
