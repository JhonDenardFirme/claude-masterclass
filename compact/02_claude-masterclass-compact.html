Module Introduction

Volume 2 goes beyond the platform into the craft layer: why prompts underperform, eight structural fixes, and how to extend those fixes into autonomous scheduled systems, visual generation, and model-selection mastery. The through-line: Claude performs at a dramatically higher level when treated as a specification executor, not a search engine.

A A. The Five Prompt Mistakes A
A1Searching Instead of Briefing

Claude generates from your specification — it is not retrieving a pre-written answer. Keyword-style queries ("write article about nootropics") give it no audience, no angle, no constraints — producing generic Wikipedia-level output. Stop searching, start briefing. Source benchmark: brief-style prompts → 3× better first drafts, 75% fewer revisions across 50 tests. Anthropic frames it explicitly: Claude is a new employee who performs as well as the brief it receives.

›Write a technical spec, not a Google query. Precision in → precision out.
A2Wall-of-Text Prompts

When instructions, context, and constraints merge into one paragraph, Claude guesses at priorities and averages out — consistently missing buried constraints. A 300-word wall prompt missed a key "no pricing mentions" constraint in line 12; the same prompt with visual hierarchy succeeded instantly. Anthropic benchmark: 40% better instruction-following with structured prompts. Fix: line breaks + section labels separating context, task, and constraints.

›Use the line breaks the interface gives you. Context / Task / Constraints — always separated.
A3No Role or Audience

Without a role, Claude writes for no one — producing text with no voice, no depth, no vocabulary register. Role adoption is holistic: Claude picks up tone, assumed reader knowledge, vocabulary, and depth simultaneously. Tested: same medical topic, 10 role combinations — range from 6/10 (generic "doctor") to 9/10 ("interventional cardiologist writing for American Heart Journal"). Always open with You are... and You're writing for...

›Role specificity is the highest-leverage single variable in any prompt.
A4Expecting One-Shot Perfection

Anthropic's own documentation states: iteration is part of the process, not a sign of failure. Tracked usage data: first attempt 70% satisfaction → after 1 revision 92% → after 2 revisions 98%. Total time: ~5 minutes vs. 30 minutes writing from scratch. Correct workflow: draft → critique → revise. The chat continues after each response for a reason — use it.

›One-shot perfection is a myth. Two targeted revisions reach 98% satisfaction.
A5Underfeeding Context

Claude 4.6 supports 200K tokens (~500 pages) and now tracks remaining context in real time. Feeding fragments instead of full documents is driving a Formula 1 car in first gear. Example: 2-page executive summary → generic insights. Full 50-page report → Claude surfaced buried supply-chain risks in footnotes and cross-department patterns the user had missed entirely. Check the token counter — if you're at 5K of 200K, you're wasting 97.5%.

›Always load the complete document. Let Claude see the full picture.
B B. Eight Power Techniques B
B1Technique 1 — Four-Block Formula

The 2026 Anthropic-recommended standard: four blocks, each doing one job. Instructions (what to do and how) · Context (background, documents, constraints) · Task (one specific deliverable) · Output Format (exact structure). Eliminates guessing — Claude executes specified constraints instead of defaulting to generic safe mode. Mental model: you're drafting a contract, not a chat message. Clearer terms → more reliable execution.

›Task block = exactly one deliverable. Multiple asks in Task = mixed results.
B2Technique 2 — XML Tags

Claude is trained on XML — Anthropic uses it extensively internally. Wrap sections in <instructions>, <context>, <examples>, <constraints>, <output_format>. Benchmark: 94% formatting compliance with XML vs. 61% without — 33pp improvement. XML is Claude's folder system for input: same content, radically different navigability.

›XML implements the four-block formula formally. These two techniques always pair.
B3Technique 3 — Prefill (Response Seeding)

Write the first words of Claude's response yourself — Claude continues from exactly that point, skipping preamble entirely. Applied to 50 product descriptions: without prefill, 47 words of "Sure, I'd be happy to help" before content; with prefill, clean consistent bullet structure for all 50 items — 2 hours of reformatting reduced to 15 minutes. Best for: rigid formats (JSON, CSV, tables, lists), bulk generation, forcing a specific tone from word one.

›XML = input structure. Prefill = output start. Stack both for two layers of precision.
B4Technique 4 — Constraint Stacking

More constraints → better writing, not worse. Constraints free Claude from generic safe mode; without them, Claude defaults to "in today's rapidly evolving landscape." Stack specific rules (real stat opener · under 150 words · no jargon · one concrete example · end with a question) and Claude is forced to find original phrasing because every cliché is ruled out. Benchmark: constraint-stacked posts → 3× higher engagement over 50 posts, 3 weeks. Specificity forces originality.

›Three layers: XML (input) + prefill (output start) + constraints (output quality). One workflow.
B5Technique 5 — Prompt Chaining

Break complex multi-deliverable tasks into sequential steps where each response feeds the next. Claude's 200K context means the full chain stays coherent — step-one instructions carry full weight at step five. Example: one mega-prompt for a 5-email cold outreach sequence → five identical-sounding first emails. Four-step chain (map objections → sequence by buyer journey → write emails 1–2 each addressing one objection → complete 3–5 with building narrative) → a real sequence where each email earns the right to the next.

›Each step = one focused deliverable. Review before continuing — this is the quality checkpoint.
B6Technique 6 — One Style Example

Drop one well-chosen writing example into the prompt instead of describing style with adjectives. Claude mirrors voice, rhythm, paragraph length, sentence structure, and punctuation habits from a single paragraph — adjectives like "friendly but professional" are contradictions Claude averages into mush. Tested across 30 styles: one example consistently outperformed multi-paragraph style descriptions. Prompt structure: "Here's how I write: [example]. Match this style exactly. Now write: [task]."

›Show, don't describe. One concrete example > any number of style adjectives.
B7Technique 7 — Motivated Instructions ("Because" Rule)

Adding because and a rationale to any rule causes Claude to apply it more precisely — a documented Claude 4.6 behavior. "Don't use ellipses" = command (follows the letter). "Don't use ellipses because the output will be read by TTS and the system can't pronounce them" = motivated rule (follows the spirit, handles edge cases). Especially powerful for non-obvious formatting, content restrictions, and structural constraints.

›Command → letter of the rule. Motivated instruction → spirit of the rule in every edge case.
B8Technique 8 — Iterative Refinement

Formal three-step process: (1) generate first draft with a well-structured prompt; (2) ask Claude to critically analyze its own response (weak spots, redundancies, gaps, contradictions); (3) ask Claude to rewrite incorporating the critique. One model, two roles: author and editor. Benchmark with full technique stack: 1,500-word publishable guide in 12 minutes. Anthropic's prompt console has a built-in prompt improver automating a 4-step variant of this process.

›Step 2 prompt: "Critically analyze your response. Find weak spots, redundancies, gaps, contradictions."
B9Prompt Debug Protocol (4 Steps)

When output is still wrong: 1. Reread as a stranger — half of problems are instructions clear in your head, ambiguous on screen. 2. Add explicit "don't do X, do Y instead" constraints block. 3. Add "if data is insufficient, say so, don't guess" — one line that dramatically reduces hallucinations. 4. Stop describing the format; show the exact output schema (JSON skeleton or headed structure) using prefill.

›One small fix often flips the entire output. Debug the prompt before blaming the model.
C C. Claude Routines C
C1What Routines Are

Claude Routines (released April 14, 2026 — research preview) are autonomous scheduled tasks running local or cloud. Key distinction from N8N/Zapier: automation tools are deterministic (same input → same output, no judgment). Routines are reasoning-based — Claude adapts if inputs change, making judgment-requiring tasks (email triage, PR review) automatable for the first time. Access: claude.ai/code/routines or desktop Code tab → Routines.

›Routines unlock automation for tasks that require judgment, not just data movement.
C2Local vs Remote

Local: 4 fields (name, description, instructions, schedule), 2-minute setup, zero infrastructure — but only runs while machine is awake and Claude Code is open. Remote: runs on Anthropic's cloud (fresh Linux · 4 vCPUs · 16 GB RAM · 30 GB disk per run), laptop-off guarantee — but requires a GitHub repo (empty private repo, 30-second setup), environment variables for secrets, and cloud connectors. Strategy: start local, graduate to remote when you need the laptop-off guarantee or connector integrations.

›Local = prove the logic. Remote = productionize it. Same instruction paragraph, different engine.
C3Trigger Types

Schedule: hourly to weekly; minimum interval 1 hour (no sub-hourly). API trigger: dedicated HTTPS endpoint + bearer token per routine; fires on POST; required beta header: anthropic-beta: experimental-cc-routine-20260401. Any tool that sends HTTP (Zapier, N8N, form submissions, Slack commands) can trigger a routine. GitHub Events: PR opened/updated, release published; granular filters by author, branch, labels, draft status. Critical: events past hourly cap are silently dropped (not failed, not logged) in current preview.

›Sub-hourly needs? Use API trigger + external cron. N8N + Routines complement each other.
C4Environment Variables, Connectors & The Env Trap

Never put secrets in .env for remote routines. .env is gitignored → never reaches GitHub → cloud environment never sees it → routine fails silently or behaves unpredictably. Secrets go in the Environment Variables panel inside Cloud Environment. Always add to your prompt: "API key is in an environment variable — do not look for a .env file." Without that line, Claude may try .env by habit. Cloud connectors (Slack, Gmail, GitHub, Notion, Linear, Drive, etc.) are authorized in the connector directory — local MCP servers are unreachable from cloud routines.

›Env trap is the #1 failure pattern. Secrets in Environment Variables panel. Always. Non-negotiable.
C5Session Log

After each run the session log (right panel) records every step, tool call, file read/write, decision, timestamp, and token count. Read it like a PR diff — it is both debugging surface and run receipt. Simple routine run time: 2–4 minutes. Production Gmail + Slack routine: ~4 min 11 sec.

›If output is wrong, the log shows exactly which step deviated and why.
C6Seven Gotchas

1. Env trap — secrets in panel not .env. 2. Network trust — start trusted mode; allowlist custom domains only when needed. 3. 1-hour floor — no sub-hourly scheduling; use API trigger. 4. Statelessness — every remote run starts from zero; persist state to GitHub, a database, or a connector. 5. claude/ branch guard — Claude can't push to main by default; leave this on. 6. Identity scope — everything runs under your account (emails, commits, Slack messages). 7. No mid-run approval — Claude runs start to finish; for approval gates, build a two-routine architecture (Routine A drafts → human approves → Routine B executes via API).

›Gotchas 1, 4, and 7 cause the most lost hours. Internalize these three before building anything.
D D. Claude Design D
D1What Claude Design Is

Claude Design lives at claude.ai/design. Not an image generator — a brand-consistent visual production environment for landing pages, decks, prototypes, carousels, and one-pagers without Figma or design skill. Framing: a junior designer who already read your brand book. Four modes: prototype · slide deck · from template · other. Always use prototype + high-fidelity — wireframe is for rough exploration only.

›Output matches your team's aesthetic, not a default AI style — because of the design system you define.
D2Design System Setup (Most Critical Step)

Never skip the design system and hit "create" immediately — that's the mistake that produces generic AI aesthetics. One field: brand description + colors + fonts + voice + references, all together, 200-word minimum. Hack: getdesign.md provides brand system markdown files from Vercel, Linear, Stripe, Notion, and more — copy the closest match, save as .md, drag into the field. After building, approve each section (typography, colors, spacing, components), publish, and set as default — every new project inherits it automatically.

›Tight design system = shorter prompts = fewer tokens per generation. Push work upstream into setup.
D3Three Refinement Modes

Exhaust all three before touching the main prompt. Tweaks: auto-generated design controls (atmosphere, glow, accent color, spacing rhythm, motion toggles) — costs zero tokens. Markup: click any element or drag to draw around a section, then give targeted feedback directly — fastest method for specific changes. Edit: direct property panel (background color, font, size) — no Claude involved, no token cost. Priority: Tweaks → Markup → Edit → main prompt. Most iterations should happen in the first three.

›Tweaks is a free preview gallery. Always run it before spending tokens on regeneration.
D4Token Rules & Export Pipeline

Three token rules: 1. Never iterate on a full page — regenerate a single section (full page costs ~10× a section). 2. Batch related work in one session — switching projects burns extra tokens per restart. 3. Tight design system enables short prompts — fewer input tokens per run. Export pipeline: Share menu → ZIP / PDF / PPTX / standalone HTML / Canva / hand off to Claude Code. Hand off → Claude Code reads the design file and builds the code → push to GitHub → deploy to Vercel → live URL in ~10 minutes. Always add a README before exporting: one extra prompt, 10× smoother handoff.

›Full pipeline: Claude Design → Claude Code → GitHub → Vercel → live in ~10 min.
E E. Claude 4.5 Model Family E
E1Architecture Changes

Claude 4.5 rebuilt four foundations: 200K context across all three models (Sonnet also has 1M beta). Extended Thinking on all three models including Haiku. Context awareness: models track remaining tokens after every tool call — prevents premature mid-task abandonment (a known prior failure). Memory tool: external storage for long-running projects — stores key info in a file, retrieves when needed, without consuming context window on every call.

›Claude 4.5 is built for sustained work — sessions stay coherent longer than any prior version.
E2Sonnet 4.5 — Daily Driver

Anthropic's "best model for agents and everyday power tasks." SWE-Bench Verified: 77.2%. Terminal Bench 2.0: 40–46% in published runs. Pricing: $3/M input · $15/M output. Handles writing, research, data analysis, brainstorming, project planning, and coding smoothly at mid-tier cost. Default in the Opus→Sonnet Claude Code handoff (plans with Opus, builds with Sonnet).

›Start here. Escalate to Opus only when Sonnet's depth is genuinely insufficient.
E3Opus 4.5 — Expert Reviewer

Flagship model. SWE-Bench Verified: 80.9% — first to break 80%. Terminal Bench 2.0: high 50s–low 60s. Pricing: $5/M input · $25/M output. Exclusive feature: effort parameter — control thoroughness per response (high / medium / low). One model that adapts to the task instead of requiring model switches for different depth levels. Reserve for: final reviews, costly decisions, deep research, complex multi-agent coordination.

›Effort parameter is Opus-exclusive. High effort = max thoroughness. Low effort = quick answers.
E4Haiku 4.5 — Speed & Scale Engine

Near-frontier performance at 1/3 the cost and 4–5× the speed of Sonnet. SWE-Bench Verified: 73.3%. Pricing: $1/M input · $5/M output. Trade-off: loses context in long sessions (may forget variable names, change identifiers). Developer pattern: Sonnet orchestrates → multiple Haiku instances execute subtasks in parallel. Best for: support bots, content moderation, high-volume document processing, sub-agent execution, rapid prototyping.

›Keep Haiku tasks short and well-defined. For extended back-and-forth, switch to Sonnet.
E5Extended Thinking in Claude 4.5

Produces a visible thinking block before the response — a summarized reasoning trace showing which approaches were considered, rejected, and selected. Key behavior: interleaved thinking — Claude reasons between tool calls, enabling coherent multi-step agentic loops. Operational note: thinking blocks change every run → reduces prompt caching efficiency. For repetitive identical workflows, toggle Extended Thinking off to preserve caching benefits.

›Thinking is summarized (not raw CoT). Expand it — learn how Claude structures problems.
F F. Claude 4.5 vs Gemini 3 Pro F
F1Where Gemini 3 Pro Leads

General reasoning: 91.9% GPQA Diamond vs. Opus ~87%. Multimodal: 87.6% Video-MMU (video understanding) — Claude doesn't compete in video. Native 1M token context vs. Sonnet's 200K default. Cost: $2/M input · $12/M output — cheaper than Sonnet ($3/$15) and far cheaper than Opus ($5/$25). For high-volume inference where accuracy margin is less critical, Gemini is more cost-efficient.

›Gemini wins: video · multimodal · raw context · cost. Claude cannot generate images or process video.
F2Where Claude 4.5 Leads

Code reliability: Sonnet 77.2% / Opus 80.9% vs. Gemini 76.2% SWE-Bench. Developer report: Claude's code "just works more often" — fewer hallucinated functions, fewer broken imports. Agent coherence: for long-running multi-tool-call tasks, Claude stays on track longer. Developer verdict: Gemini = brilliant but unpredictable; Claude = dependable. Claude exclusives: Extended Thinking (transparent reasoning) + effort parameter (thoroughness control). Neither has a Gemini equivalent.

›Claude wins: code · agent reliability · transparency · regulated industries · mission-critical work.
F3Decision Matrix

Use Gemini 3 Pro: video/multimodal work · need largest native context · cost-optimizing high-volume inference · comfortable supervising a powerful-but-unpredictable assistant. Use Claude 4.5: reliable code + agent workflows · transparency matters · regulated industries · mission-critical correctness · want a "careful senior engineer" over a "fast-moving junior." Production split recommended by the source: Gemini for front-end visuals and multimodal; Claude for back-end logic, data analysis, and anything where mistakes have consequences. These are complements, not competitors.

›Optimal stack uses both. Gemini for visuals; Claude for everything where correctness has consequences.
Module Summary
  • Five mistakes share one root: treating Claude as a search engine. Fix all five simultaneously — brief-style prompts, visual structure, specific role, iterative mindset, full context — and the improvement is compound.
  • Eight techniques build in order: four-block formula → XML → prefill → constraint stacking → chaining → style example → motivated instructions → iterative refinement. Stack all eight for complex high-stakes tasks; pick the relevant subset for simpler work.
  • Routines: start local (4 fields, 2 minutes), graduate to remote when you need cloud execution or connectors. Non-negotiable: secrets in Environment Variables, not .env. Know the seven gotchas before building.
  • Claude Design: design system first, always. Minimum 200-word brand paragraph. Three refinement modes (Tweaks → Markup → Edit) before the main prompt. Never regenerate full pages — section-level iteration costs 10× less.
  • Model decision: Sonnet (daily driver, $3/$15) → Opus (final review + effort parameter, $5/$25) → Haiku (sub-agent executor, $1/$5). Extended Thinking available on all three; caching trade-off applies.
  • Claude vs. Gemini: not a competition — complementary tools. Gemini for video/multimodal/cost; Claude for code reliability, agent coherence, and correctness-critical work.
